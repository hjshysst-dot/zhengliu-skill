# 胖叔蒸馏法 PANGHSU Zhengliu Method

> 融合 Cangjie(书蒸馏) + Nuwa(人蒸馏) + Darwin(自动进化) 的三元统一框架。

## 核心理念

**不是书摘，不是角色扮演，是可执行的认知操作系统。**

输入：知识载体（书/人/课程）→ 输出：原子化、可触发、可测试、可进化的技能包。

三个维度缺一不可：
- **Cangjie 维度**：从知识载体中提取方法论框架
- **Nuwa 维度**：捕捉决策思维（How they think / Why they decide）
- **Darwin 维度**：内置压力测试和进化机制，确保长期有效

---

## 解决原版 Cangjie 的 8 个弊端

| # | 原版弊端 | 胖叔版解决方案 |
|---|---------|--------------|
| 1 | 三重验证主观性 | **增加第4验证：自我一致性(V4)** —— 同一方法论被多个extractor提取时，交叉比对消除歧义 |
| 2 | 5个并行提取器缺乏协调 | **增加协调层(Coordinator)** —— 合并重复、仲裁冲突、淘汰冗余 |
| 3 | 诱饵测试鸡生蛋 | **改为"边界探测测试"** —— 问"这个skill在什么情况下会失败"比"什么情况下不该调用"更可操作 |
| 4 | V3独特性过于严格 | **增加"应用密度"指标** —— 有实践价值的内容即使不够"反直觉"也可通过 |
| 5 | 格式依赖性强 | **增加格式质量检测** —— OCR错误率、翻译偏差检测、源头质量评分 |
| 6 | 维护成本高 | **自动化INDEX更新** —— skill更新时自动触发相关skill引用图重建 |
| 7 | INDEX链接过时 | **版本锁定+变更日志** —— 每次更新记录影响范围，支持按版本回滚 |
| 8 | 原子化边界模糊 | **建立原子化标准** —— 明确"1个触发条件=1个skill，触发条件定义到具体场景" |

## 执行流程

### 阶段 0 — 知识理解

**知识类型判断**：

| 类型 | 来源 | 采集策略 |
|------|------|---------|
| 书 | PDF/EPUB/TXT | Adler四步 + 5维度并行提取 |
| 人 | 访谈/著作/transcript | Nuwa 6维度并行采集 |
| 课程 | 视频/字幕/讲义 | 拆解为"书+人"混合处理 |
| 混合 | 多源 | 按类型分流后合并 |

**Adler 四步（书）**：
1. 结构：章节骨架、核心论点分布
2. 解释：关键概念、术语体系
3. 批判：作者的盲点、局限性
4. 应用：真实场景中如何调用

**6维度并行采集（人）**：
1. 著作：系统性观点、反复出现的论点
2. 对话：即兴反应、类比模式
3. 表达：高频用词、风格DNA
4. 决策：决策逻辑、边界案例
5. 时间线：思想演变、最近动态
6. 他者：外部评价、批评争议

**源头质量评估（Darwin维度）**：
```
质量检查项：
- 格式完整度（1-10）：是否完整章节/章节缺失/大量OCR错误
- 翻译质量（1-10）：术语准确性、表达流畅度
- 一手vs二手（1-10）：直接来源vs转述占比
- 时间新鲜度（1-10）：内容是否过时（>10年需特别标注）

源头质量分 = 加权平均(格式×0.2 + 翻译×0.2 + 一手×0.4 + 新鲜×0.2)

源头质量分 < 5分的书 → 警告用户，建议更换来源或降低预期
```

**QA清单生成（Nuwa维度）**：
针对该知识载体，生成"这个人在XX问题上可能会说什么/避免什么"：
- 防忽悠测试问题（用于阶段4）
- 常见误解预警

### 阶段 1 — 协调提取（改进版）

**5个并行Extractor**：

| Extractor | 提取目标 | 输出文件 |
|-----------|---------|---------|
| 框架 | 决策框架/思维模型 | candidates/frameworks.md |
| 原则 | 原则/清单/规则 | candidates/principles.md |
| 案例 | 作者亲自用过的实例 | candidates/cases.md |
| 反例 | 失败模式/警告 | candidates/counter-examples.md |
| 术语 | 关键概念词典 | candidates/glossary.md |

**Coordinator 协调层（新增）**：

并行提取完成后，自动执行：

```
1. 去重合并：
   - 同一方法论被多个extractor提取 → 合并为1条，标注来源
   - 术语冲突（不同名字同一概念）→ 建立同义词表

2. 一致性仲裁：
   - 框架说"X是Y"但案例说"X不是Y" → 记录为"待验证矛盾"
   - 仲裁规则：原文引用优先；无原文时保留矛盾让用户判断

3. 冗余淘汰：
   - 描述完全重复的候选 → 合并
   - 与其他候选包含关系（一个是另一个的子集）→ 拆分为独立skill

4. 输出：
   - candidates/merged.md：去重合并后的候选池
   - candidates/consistency-report.md：仲裁记录和矛盾清单
```

---

### 阶段 1.5 — 四重验证（升级版）

#### V1 跨域验证
同原版：书中至少2个独立语境有佐证

#### V2 预测力验证
同原版：能用它推导出书里没明说的新问题吗？

#### V3 独特性验证（改进）
```
改进：增加"应用密度"豁免条件

V3通过标准（二选一）：
A. 不是常识（原有标准）
B. 有高应用密度：能在≥3个不同场景中被实际调用

应用密度评估：
- 列出该方法论可以实际解决问题的5个具体场景
- 能列出≥3个 → 通过V3（即使不够反直觉）
- 只能列出0-2个 → 需要满足原有"非常识"标准

原因：实践价值是skill的核心价值，反直觉但无用的内容不值得skill化
```

#### V4 自我一致性验证（新增）
```
问题：这个skill内部是否自洽？

检查项：
1. R(原文)和I(重写)是否一致？不能矛盾
2. E(执行步骤)是否能还原出I(方法论)？不能脱节
3. B(边界)和A2(触发)是否互斥？不能边界=触发
4. 与同一知识载体的其他skill是否冲突？不能重叠定义

通过：打勾
不通过：记录为"自洽性问题X级"，进入重做
```

#### 格式鲁棒性检测（新增）
```
OCR质量检查：
- 错字率 > 5% → 警告，降低该候选优先级
- 关键术语缺失/模糊 → 记录为"需要补充定义"

翻译偏差检测：
- 核心术语是否走味 → 记录为"翻译风险点"
```

**验证结果格式**：
```yaml
id: f01
title: 逆向思维
type: framework
V1_cross_domain: { passed: true, evidence: [章节3投资, 章节7工程, 章节11教学] }
V2_predictive_power: { passed: true, novel_question: "...", derived: "..." }
V3_exclusivity: { passed: true, via: "B", 应用密度场景: 5 }
V4_self_consistency: { passed: true }
format_quality: { score: 8, issues: [] }
→ 进入阶段2
```

### 阶段 2 — RIA++E 构造（增强版）

**每个通过四重验证的单元，构造完整skill**：

```
R (Reading): 原文引用 ≤150字/段，标注章节页
I (Interpretation): 用自己的话重写骨架，不照搬译本
A1 (Past Application): 作者在书中亲自用过的案例
A2 (Future Trigger) ★: 具体场景问题——"在什么情况下你会调用这个skill"
E (Execution): 1-2-3可执行步骤（必须可操作，不能是空话）
B (Boundary): 什么时候不适用 + 作者的盲点

【新增】
Trigger量化：
- 输入：用户会说的具体话（≥3个变体）
- 输出：明确的触发条件定义
- 示例："当用户说'这件事能不能做'且涉及商业决策时 → 调用此skill"

应用密度评分（1-5）：
- 5：日常高频场景
- 4：专业场景，月均用1次
- 3：专项场景，年均用几次
- 2：边缘场景，几年1次
- 1：极端边缘
```

### 阶段 3 — 达尔文进化（内置版）

#### 3.1 边界探测测试（替代诱饵测试）

```
改进理由：问"什么情况下不该调用"太抽象。
边界探测更可操作：问"这个skill在什么情况下会失败"。

探测问题设计（每个skill至少5条）：
1. 最典型的失败场景是什么？
2. 超出边界时skill会给出什么错误输出？
3. 用户最常误解的是什么？
4. 和哪些skill容易混淆？区别是什么？
5. 在什么条件下应该切换到另一个skill？

示例（"逆向思维"skill）：
- 失败场景：当问题无法反转时（如数学证明题）
- 错误输出：给出反向但同样错误的答案
- 误解：以为"反向=正确"
- 易混淆：与"第一性原理"混淆
- 切换条件：问题结构已固定时
```

#### 3.2 测试prompt设计

```
测试集结构（test-prompts.json）：
{
  "should_trigger": [
    { "prompt": "用户会说的话", "expected": "skill应该输出什么" }
  ],
  "boundary_probes": [
    { "scenario": "边界场景描述", "expected_behavior": "应该拒绝/警告/降级" }
  ],
  "should_not_trigger": [
    { "prompt": "不该调用的情况", "expected": "skill应该识别并拒绝" }
  ]
}

数量要求：
- should_trigger: ≥3条
- boundary_probes: ≥3条（来自边界探测结果）
- should_not_trigger: ≥2条
```

#### 3.3 评分rubric（Darwin 8维度本地化）

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| Frontmatter质量 | 8 | name规范、description含触发条件、描述≤200字 |
| 工作流清晰度 | 12 | 步骤明确、E步骤可操作、有边界识别 |
| 边界条件覆盖 | 12 | B段完整、边界探测覆盖≥5场景 |
| 触发条件量化 | 15 | A2具体可执行、Trigger变体≥3 |
| 指令具体性 | 12 | 无模糊词、有示例、有格式模板 |
| 自我一致性 | 10 | V4通过、结构自洽 |
| 应用密度 | 10 | 评分≥3（过低说明价值不足） |
| 实测表现 | 21 | 边界探测验证 |

**总分 = Σ(维度分×权重)/100，满分100**

#### 3.4 进化循环

```
for each skill:
  current = 基线评分
  
  for round in 1..3:
    诊断最低分维度
    生成1个具体改进方案
    执行编辑 + git commit
    重新评分
    
    if 新分 > 当前分:
      当前分 = 新分
      记录为"keep"
    else:
      git revert
      记录为"revert"
      break

  展示改动摘要 + 分数变化
  等用户确认
```

### 阶段 4 — 自动化维护系统

#### 变更传播机制

```
当某个skill更新时：
1. 扫描INDEX.md，找出引用该skill的其他skill
2. 对每个被影响的skill：
   - 检查是否需要同步更新（如边界条件变了）
   - 生成impact-report.md
3. 如果有冲突，暂停等待人类判断
```

#### 版本锁定

```
META.md中记录：
- skill_versions: {skill-name: version-hash}
- last_updated: ISO timestamp
- update_reason: 为什么改

变更时：
- 生成changelog.md记录diff
- 保留最近3个版本快照
- 可按版本回滚
```

#### INDEX自动重建

```
每次skill更新后：
1. 重新扫描所有skill的相互引用
2. 更新mermaid引用图
3. 重建skills列表（含应用密度评分）
4. 标注需要人工审核的变更
```

## 质量红线

1. 每个skill必须通过**全部四重验证**
2. 每个skill必须有完整的 R/I/A1/A2/E/B六段
3. 触发条件必须量化（Trigger变体≥3）
4. 边界探测覆盖≥5场景
5. 应用密度评分≥3
6. 自我一致性检查通过
7. 原文引用≤150字/段
8. 源头质量分<5分的知识载体必须警告用户

## 调用惯例

1. **永远先试点** —— 1本书或1个人验证全流程
2. **阶段间主动汇报** —— 不要静默跑完再dump
3. **不凭记忆蒸馏** —— 没文本/素材就停下来要
4. **保留审计轨迹** —— candidates/、rejected/、evolution-log/必须留
5. **应用场景优先** —— 蒸馏前先问用户用来解决什么问题，决定过滤标准
6. **源头质量前置** —— 阶段0必须做质量评估，差的来源降低预期
